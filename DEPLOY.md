# Deploy — aleph01.com (VPS / Plesk)

The site is a **static Vite build**. Source lives in this repo; the servable
output is generated by `npm run build` into `dist/`. `start.js` is a tiny,
zero-dependency Node server that serves `dist/` — so you never have to move
files into the web root by hand again.

## Daily change flow (fast path)

Once set up (below), pushing a change and updating the live site is:

```bash
# on the VPS, in the repo directory
npm run deploy      # = git pull && npm ci && npm run build
# then restart the Node app (Plesk button, or `node start.js` if run manually)
```

That's it — no copying `dist/` into the root. The server always serves the
freshly built `dist/`.

---

## Option A — Plesk Node.js app (recommended, matches "repo già collegata")

Plesk runs Node apps via Passenger. You already connected the Git repo; add the
Node app on top of it.

1. **Domains → aleph01.com → Node.js**
   - **Application Root**: the folder where the repo is checked out
   - **Application Startup File**: `start.js`
   - **Application Mode**: `production`
   - Node version: 18+ (three.js/Vite need a modern Node)
2. Click **NPM install**, then run **`npm run build`** once (Plesk "Run script"
   → `build`) so `dist/` exists.
3. **Enable** the app. Passenger sets `PORT` for you; `start.js` reads it
   (`process.env.PORT`) and serves `dist/`. Plesk's nginx sits in front and
   handles TLS + gzip.

### Auto-deploy on push (Plesk Git)

In **Git → Pull Updates / Deploy actions**, enable automatic deployment and set
the deploy actions to:

```bash
npm ci
npm run build
```

Then, after each pull, Plesk rebuilds `dist/` automatically. Restart the Node
app so it picks up the new build (Plesk can be configured to touch
`tmp/restart.txt`, or restart the app in the Node.js panel).

> Why a Node app instead of pointing the docroot at `dist/`? Either works for a
> purely static site, but the Node app means the **build output is decoupled
> from the web root** — the repo can stay checked out cleanly and you never
> hand-move files. It also leaves room to add real server logic later.

---

## Option B — Static docroot (no Node process)

If you'd rather not run a Node process, point the domain's **Document Root** at
the repo's `dist/` folder and keep the Git deploy actions (`npm ci && npm run
build`). nginx serves `dist/` directly. Simpler, but you lose `start.js` and any
future server-side logic.

---

## Manual / one-off deploy (no Plesk automation)

```bash
git pull
npm ci
npm run build
node start.js        # serves dist/ on $PORT (default 3000)
```

Put it behind nginx (proxy_pass to the Node port) or run `npm run build` and
serve `dist/` with any static server.

---

## Notes

- `dist/` is **git-ignored** — it's always rebuilt on the server, never
  committed. Don't commit build output.
- `start.js` caches hashed assets (`immutable`, 1 year) and keeps `index.html`
  `no-cache`, so a new deploy is visible immediately without stale HTML.
- Hiding/showing page sections is done in `src/components/Content.jsx`
  (`defaultSections`) — no build config needed, just edit and rebuild.
